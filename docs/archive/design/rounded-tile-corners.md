# Rounded Tile Corners

## Goal
Give jigsaw pieces a soft, rounded-corner look while their puzzle is still being assembled, and snap that rounding away the instant the puzzle is solved. This makes unsolved pieces read visually as discrete "tiles," while a just-solved arrangement reads as a clean, seamless image at the exact moment the existing fade-to-vanish begins. Implemented as a LÖVE2D pixel (fragment) shader, wired through the `sprite.shader` field that already exists in `lua/core/sprite.lua` but has never been used by anything in the codebase.

Additionally, the completed-puzzle "trophy shelf" image (the permanent, accumulating display of fully-solved puzzles — see `docs/archive/design/completed-puzzle-display.md`) also gets rounded corners, at the same fixed ~8px radius, but **permanently** — this rounding is never removed, since a shelf entry never transitions between "unsolved" and "solved" states the way a piece does. It's a decorative frame on the finished image, conceptually separate from the piece-rounding-removal mechanic even though it reuses the same shader source.

## Affected files
- `assets/shaders/rounded_corners.frag` — **new**. GLSL fragment shader shared by both pieces and shelf entries. Radius is a hardcoded `const float RADIUS = 8.0;` (fixed everywhere per decision — never varies, never sent). Size is a `uniform vec2 size;` (in pixels), because it's the one quantity that differs between the two use sites: pieces are always `C.SLOT` × `C.SLOT` (64×64), but shelf images vary per puzzle (`cols * C.SLOT` × `rows * C.SLOT`). Pixels outside the rounded-rect footprint (computed from `texture_coords * size` against `RADIUS` per corner) get alpha 0; pixels inside pass the sampled texel, multiplied by the incoming vertex color, through unchanged (so the existing fade-to-vanish alpha and any tint still work normally).
- `game/jigsaw_piece.lua`:
  - At module scope, load the shader once via `lua/core/shader.lua`'s existing `load(path)` helper, then immediately call `piece_shader:send("size", {C.SLOT, C.SLOT})` once — this value never changes, so it's sent exactly once for the module's lifetime, not per piece or per frame. The compiled shader is then shared (assigned by reference) across every piece.
  - `JigsawPiece.new`: when constructed with a `visual` (image + quad — i.e. real ejected pieces, not the bare test-only pieces some tests build without a `visual` table), also assign the shared shader to `self.sprite.shader`.
  - `JigsawPiece:start_vanish()` — the existing "this piece's puzzle just solved" hook, already called by `game_scene.lua` the instant `JigsawSolver.is_assembled` first returns true for a puzzle — gains one added line: `self.sprite.shader = nil`. This clears the rounded-corner effect the same frame the 0.5s fade begins.
- `lua/core/shader.lua` — no change; existing `load(path)` wrapper is reused as-is (called once for the shared piece shader, and again for each new shelf entry — see below).
- `lua/core/sprite.lua` — no change; `self.shader` and the `setShader`/unset-around-draw logic already exist and already work for whatever shader object is assigned to a sprite.
- `game/scenes/game_scene.lua` — **now does change**, for the shelf mechanic only (pieces still require zero changes here, as originally designed — the shader-clear stays folded into `start_vanish()`). In the fade-prune loop that builds a `shelved = {...}` drawable once a solved puzzle's pieces finish fading (the block that constructs `image`/`x`/`y`/`cols`/`rows`/`draw`), compile a fresh shader instance via `Shader.load(...)`, send its `size` uniform once with that puzzle's actual pixel dimensions (`cols * C.SLOT`, `rows * C.SLOT`), and store it on the `shelved` table (`shelved.shader = ...`). The `draw` closure wraps its existing `love.graphics.draw(self.image, self.x, self.y)` call with `love.graphics.setShader(self.shader)` / `love.graphics.setShader()`, mirroring the pattern `Sprite:draw()` already uses. This shader instance is never cleared — it's compiled once per completed puzzle (an infrequent event, bounded by how many puzzles the player solves in a session) and persists for the shelf entry's permanent lifetime.
- `lua/headless/stubs.lua` — **new stub needed**. Today `love.graphics.newShader` isn't explicitly stubbed; it falls through the generic `new*`-prefix catch-all in `graphics_stub`'s `__index` metamethod and resolves to `make_stub_image`, whose returned table has no `send` method. Calling `:send(...)` on it (now required by both the piece module and the per-shelf-entry code) would error under `--headless`. Add an explicit `graphics_stub.newShader = function(path) return { send = noop } end` alongside the other explicit stubs (e.g. next to `setShader`) so it takes priority over the catch-all and `:send()` becomes a safe no-op in test mode, matching the file's existing "explicit stub first, generic catch-all fallback" convention.
- `tests/test_jigsaw.lua` — new coverage: a freshly constructed piece (with `visual`) carries a non-nil `sprite.shader`; calling `start_vanish()` clears it to `nil`; a shelved trophy entry (once a puzzle fully fades) carries a non-nil `shader`; existing solved-puzzle and shelf-placement assertions continue to pass unchanged. Headless-safe once the `lua/headless/stubs.lua` addition above lands — `love.graphics.newShader(...)` returns a stub with a working (no-op) `send`, so both `:send()` calls and shader assignment/clearing are exercised structurally without needing real GLSL execution.

## What changes
- Unsolved, ejected jigsaw pieces render with visually rounded corners (~8px, fixed) instead of sharp square corners.
- The instant a puzzle's `is_assembled` check first returns true, each of its pieces' shader is cleared via the existing `start_vanish()` call, so it draws sharp-cornered for whatever remains of that frame before the pre-existing 0.5s alpha fade (`C.PIECE_FADE_DURATION`) hides it. No new timing, no hold state, no easing on the radius itself — a plain instant assignment mirroring the "instant snap" decision.
- The trophy shelf's completed-puzzle images also render with rounded corners at the same fixed ~8px radius — but this rounding is applied once, at the moment a shelf entry is built, and **never cleared**. There's no "solved" transition to react to for a shelf entry (it's already the finished, permanent record of a solve), so this is a one-way, one-time visual treatment, not a mirror of the piece mechanic's snap-off behavior.
- One shader source (`assets/shaders/rounded_corners.frag`) is shared by both draw paths. The two use sites differ only in what pixel `size` they send it (fixed `{64, 64}` for pieces, sent once at module load; per-puzzle `{cols*C.SLOT, rows*C.SLOT}` for shelf entries, sent once per completed puzzle) — radius is a shader-side constant in both cases, never sent as a uniform.
- Net code diff: one new shader asset file, a small addition to `game/jigsaw_piece.lua`, a small addition to the existing shelf-building block in `game/scenes/game_scene.lua`, and one new explicit stub in `lua/headless/stubs.lua`. No other entity (box, floor, spawn button) is touched.

## What stays the same
- `JigsawBox`, `SpawnButton`, `Door`, and the floor checkerboard — all untouched, no shader applied to any of them (scope is jigsaw pieces and the trophy shelf only, per decision).
- The trophy shelf's placement logic — the deterministic left-to-right shelf-slot math (`shelf_row_x`/`shelf_row_bottom`/`shelf_row_max_height` bookkeeping), when an entry is created (once its pieces fully fade), and what image/`cols`/`rows` it carries — is entirely unchanged. Only the `draw` closure itself gains a shader wrap, and the entry gains a `shader` field; positioning, wrapping-to-a-new-row, and the underlying image are untouched.
- Puzzle-solve detection (`game/jigsaw_solver.lua`'s `is_assembled`), the 0.5s fade-to-vanish (`JigsawPiece:update_fade`, `C.PIECE_FADE_DURATION`), and the trophy-shelf hand-off timing in `game_scene.lua` — all logic and timing unchanged. The piece shader-clear is an additive side effect of the existing `start_vanish()` call, not a new hook or new state; the shelf shader is an additive attachment at the existing shelf-entry-creation point, not a new lifecycle stage.
- Piece pickup/drop/rotate and drag-ghost preview (`draw_ghost`) — unaffected. Rotation still looks correct with the shader applied: `love.graphics.rotate()` rotates the whole quad (and its rounded mask) together in `Sprite:draw()`, so rounded corners stay aligned to the piece regardless of `rotation_step`.
- Radius (~8px) is always a shader-side constant, never sent as a uniform, for either draw path — it truly never varies, so there's no reason to pay for a `send` call on it.
- Each distinct compiled shader instance only ever has one `size` value sent to it, exactly once, at creation — never resent, never overwritten. The piece-shared instance is created once (module load) with `{64, 64}` and handed out by reference to every piece, since all pieces are always that size. Each shelf entry compiles its own separate shader instance (not shared with pieces or with other shelf entries) and sends its own fixed `size` once when the entry is built. Because uniform state lives per-Shader-object and no instance is ever reused across two different sizes, there's no risk of one draw call's size leaking into another's — no per-frame resending is needed anywhere.
- `shader:send()` is now called (a fixed, small number of times — once per piece-shader load, once per completed puzzle), which is why the new explicit `newShader` stub in `lua/headless/stubs.lua` is required; everything else about the headless test environment (`setShader` as a no-op, etc.) is unchanged.

## Open questions
None — resolved during Phase 1 (scope, transition timing, radius animation, and radius size were all decided by the user before this doc was written; the trophy-shelf addition — permanent rounding, same fixed ~8px radius, not proportional to image size — was confirmed by the user as a follow-up).

Investigated and resolved as part of the trophy-shelf addition: `lua/headless/stubs.lua`'s stub shader (returned via the generic `new*` catch-all) does **not** support `:send()` — it resolves to `make_stub_image`'s table, which has no such method and would error if called under `--headless`. Resolved by adding an explicit `graphics_stub.newShader` stub (see "Affected files") rather than mutating the shared `make_stub_image` factory used by `newImage`/`newQuad`/etc., keeping the fix scoped to shaders only.

Minor, non-blocking implementation note left to the task phase: exact GLSL syntax for the rounded-rect mask (e.g. per-corner distance check vs. a single SDF formula) — the per-corner distance-check approach described above is the recommended default and is a well-known, simple pattern.
