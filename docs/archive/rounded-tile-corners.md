Task B (the `.frag` shader source) is fully independent and can be done in parallel with anything — nothing in headless test mode reads the actual file contents (the stub ignores the `path` argument), so its landing order relative to A/C/D/E doesn't matter for tests, only for real (non-headless) rendering.

- [x] Task B — `assets/shaders/rounded_corners.frag` — Create this new GLSL fragment shader file (LÖVE2D pixel shader convention: a function `vec4 effect(vec4 color, Image tex, vec2 texture_coords, vec2 screen_coords)`). Requirements: (1) declare `const float RADIUS = 8.0;` — a hardcoded constant, never a uniform, never varies; (2) declare `uniform vec2 size;` — the drawable's pixel dimensions, sent once by callers (`{64,64}` for pieces, `{cols*SLOT, rows*SLOT}` per shelf entry); (3) compute the fragment's pixel position as `texture_coords * size`; (4) for each of the four corners, check whether the pixel falls within that corner's `RADIUS`-sized square region, and if so compute its distance from that corner's rounding-circle center — if the distance exceeds `RADIUS`, the pixel is outside the rounded-rect footprint and should get alpha `0.0`; pixels not in any corner region, or within radius of the corner circle, are inside the footprint; (5) for pixels inside the footprint, sample the texture (`Texel(tex, texture_coords)`), multiply by the incoming vertex `color` (do not skip or override this multiply — it's what keeps the existing fade-to-vanish alpha and any tint working), and return that; for pixels outside, return the same texel-times-color product but with alpha forced to `0.0` (do not just discard/return black — preserve RGB, zero the alpha channel only). A standard per-corner `distance()` check against each corner's circle center (each center inset by `RADIUS` from its respective edges) is a fine, simple implementation approach. This task has no dependency on any other task and can be done in parallel with all of them.
